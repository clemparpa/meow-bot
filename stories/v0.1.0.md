# Stories — `v0.1.0` (`MENTION_REVIEW` end-to-end)

> Statut : draft v0.1 · à itérer
> Référence spec : [§17 Roadmap](../SPEC.md), [§5 Permissions](../SPEC.md), [§6 Webhook Receiver](../SPEC.md), [§7 Worker](../SPEC.md), [§8 Workflows et activities](../SPEC.md), [§9 Profils d'intent](../SPEC.md), [§11 Auth GitHub App](../SPEC.md), [§13.2 Journal d'audit](../SPEC.md), [§14 max_price/max_turns](../SPEC.md)

## Objectif de la version

Livrer la **première feature réellement utile** du bot : `@<bot-login> review` posté en commentaire d'une PR → meow-bot lit le diff dans un sandbox Daytona, exécute `mistral-vibe` en mode programmatique, et poste son rapport en commentaire. Read-only de bout en bout.

Au-delà de la feature, ce cycle introduit l'**infrastructure technique définitive** :

- `githubkit` comme SDK GitHub unique (auth App + REST + parsing webhook typé) — **retrait du module HMAC maison** livré en v0.0.x.
- `mistralai-workflows` comme orchestrateur durable (worker `run_worker(discovered)` + workflows + activities).
- Daytona pour le sandbox éphémère.
- `mistral-vibe` comme agent codeur (mode `run_programmatic`).
- `events.jsonl` append-only (§13.2).
- Quickstart 15-min documenté.

**Critère de sortie (spec §17)** : dogfood — meow-bot review ses propres PRs sans intervention manuelle pendant 1 semaine.

## Hors-périmètre v0.1.0

- Classifier Mistral Small + multi-intents → `v0.2.0`.
- Budgets par thread (SQLite) → `v0.2.0`.
- Profil `WRITE_BRANCH` + intent `CODE_REQUEST` → `v0.3.0`.
- Intent `TRIAGE` (issues) → `v0.4.0`.
- `RepoScanner` cron proactif → `v0.5.0`. (Mais `InstallLifecycle` est livré ici car requis pour réagir à l'install initial.)

## Phases de parallélisation

Le graphe se résout en 6 phases. Comme pour v0.0.x, à chaque phase les stories listées peuvent être implémentées en parallèle (typiquement : une branche `feat/sN-...` par story).

| Phase | Stories en parallèle | Bloque-passage |
|---|---|---|
| **A** (fondations) | S1 puis S2, S3 | S1 mergée avant S2/S3 ; S2 + S3 avant phase B. |
| **B** (orchestration) | S4 puis S5 puis S6 | Chaîné — S4 avant S5 (squelette a besoin de la dep) ; S5 avant S6 (le receiver enqueue dans un worker qui doit exister). |
| **C** (chaîne stubbée) | S7, S8, S9, S10 | Toutes mergées avant phase D — la signature de S9 (input/output de l'activity) doit être figée pour que S11/S12 la respectent. |

| **E** (hardening + docs) | S14, S15, S16 | S14 d'abord (`events.jsonl` instrumente tout le reste) ; S15 et S16 en parallèle après. |
| **F** (dogfood) | S17 | — (clôt v0.1.0). |

**Points d'attention quand on parallélise** :

- La phase C est la plus large (4 stories en parallèle). Bien geler la signature de `GithubEventHandler` (workflow + intent dispatch) à la fin de S7, et les types `PrContext` / `ReviewReport` aux interfaces de S8/S9/S10.
- `README.md` et la table "Vue d'ensemble" de ce document seront touchés à chaque story — garder les diffs minimes.
- Marquer le statut (`✅` / `🟡` / `⛔`) **dans la PR de la story**, pas dans un commit séparé.
- Si deux stories doivent toucher la même région d'un fichier partagé, sérialiser : faire la première, rebaser la seconde.
