# Stories — `v0.0.x` (scaffold)

> Statut : draft v0.1 · à itérer
> Référence spec : [§17 Roadmap](../spec.md), [§16 Structure du repo](../spec.md), [§6 Webhook Receiver](../spec.md), [§15 Déploiement](../spec.md)

## Objectif de la version

Mettre en place le **scaffold** : tooling Python, layout, infra Docker, manifest App, et un receiver minimal qui valide HMAC et log. Aucun appel à Mistral Workflows ni Daytona à ce stade.

**Critère de sortie (spec §17)** : `curl` d'un webhook factice signé sur le receiver → réponse `200` + une ligne de log JSON structurée sur stdout.

## Hors-périmètre v0.0.x

- Toute logique métier (parse intent, fetch PR, sandbox, post comment) → `v0.1.0`.
- Le worker `workflows.run_worker(...)` réel → `v0.1.0` (en v0.0.x, le service worker existe mais peut être un stub qui sleep).
- Tests d'intégration contre une vraie GitHub App → `v0.1.0` (dogfood).

**Points d'attention quand on parallélise** :

- `README.md`, `CHANGELOG.md` et la table "Vue d'ensemble" de ce document sont touchés par plusieurs stories — anticiper les conflits de merge en gardant les diffs minimes et localisés.
- Marquer le statut (`✅` / `🟡`) dans la table "Vue d'ensemble" **dans la PR de la story**, pas dans un commit séparé — évite la course aux updates de statut.
- Si deux stories doivent toucher exactement la même région d'un fichier partagé, sérialiser : faire la première, rebaser la seconde.

## S5 — Vérification HMAC webhook

**But** : isoler la primitive de sécurité du §12.2.

**Livrables**
- `src/meow/common/github/webhook.py` : `verify_signature(header: str | None, body: bytes, secret: str) -> None` qui raise `InvalidSignature` sinon.
- Utilise `hmac.compare_digest`.
- Gère le préfixe `sha256=` du header `X-Hub-Signature-256`.

**Acceptation**
- Test unitaire avec signature correcte → pas d'exception.
- Test unitaire signature tronquée / mauvaise → `InvalidSignature`.
- Test unitaire header `None` → `InvalidSignature`.

## S6 — Receiver FastAPI minimal

**But** : exposer `POST /gh/webhook` qui valide HMAC, filtre les self-events, log, et retourne 200. **Pas encore** d'appel à Mistral Workflows (stub : log `event: webhook.accepted` à la place).

**Livrables**
- `src/meow/receiver/app.py` : application FastAPI avec route `/gh/webhook` + `GET /healthz`.
- `src/meow/receiver/__main__.py` : entry point uvicorn.
- Filtre `sender.login == bot_login()` → log `event: webhook.skipped reason=self` et retourne `{"skipped": "self"}`. **Note** : `bot_login()` est stubbé en v0.0.x (lit env `MEOW_BOT_LOGIN` ou retourne `None`). La vraie lecture via `GET /app` arrive en v0.1.0.
- Filtre événements non gérés → `{"skipped": "event"}`.
- Réponse `<10s` (§6).

**Acceptation**
- `uv run uvicorn meow.receiver.app:app` boot sans erreur.
- `GET /healthz` → 200.
- `POST /gh/webhook` sans signature → 401.
- `POST /gh/webhook` signé, event `ping` (non géré) → 200 `{"skipped": "event"}`.
- `POST /gh/webhook` signé, event `issue_comment` → 200 `{"queued": true}` (stub) + ligne de log JSON.

## S8 — Dockerfile multi-stage

**But** : produire deux images via cibles `receiver` et `worker` (§15.1).

**Livrables**
- `Dockerfile` multi-stage :
  - stage `base` : Python 3.13-slim + `uv` + `uv sync --frozen`.
  - target `receiver` : `CMD ["uv", "run", "uvicorn", "meow.receiver.app:app", "--host", "0.0.0.0", "--port", "8000"]`.
  - target `worker` : `CMD ["uv", "run", "python", "-m", "meow.worker"]` (le module peut être un stub `while True: sleep(60)` en v0.0.x).
- `.dockerignore`.

**Acceptation**
- `docker build --target receiver -t meow-receiver .` build sans warning.
- `docker run --rm -p 8000:8000 meow-receiver` + `curl localhost:8000/healthz` → 200.
- `docker build --target worker -t meow-worker .` build et boot sans crash.
